## 3. Key Services

### 3.1 TranscriptService

Responsible for fetching and cleaning YouTube transcripts.

```php
class TranscriptService
{
    // Fetch transcript via YouTube Data API v3
    public function fetch(string $videoId): string

    // Clean raw transcript (remove timestamps, filler words, normalize)
    public function clean(string $rawTranscript): string

    // Fallback: prompt user to paste transcript manually
    public function manualFallback(): void
}
```

**Flow:**
1. Extract video ID from YouTube URL
2. Call YouTube API captions endpoint
3. If captions exist → fetch and clean
4. If no captions → attempt auto-generated transcript
5. If neither → return flag for manual paste fallback

---

### 3.2 AIAssessmentService

Core AI logic. Uses Claude claude-sonnet-4-6 via Anthropic API.

```php
class AIAssessmentService
{
    // Step 1: Extract learning objectives from transcript
    public function extractConcepts(string $cleanedTranscript): array

    // Step 2: Generate hybrid question set
    public function generateQuestions(
        array $concepts,
        string $skillCategory,
        int $questionCount = 10
    ): array

    // Step 3: Generate rubric for scoring
    public function generateRubric(array $questions): array

    // Step 4: Evaluate submitted answers
    public function evaluateAnswers(
        array $questions,
        array $rubric,
        array $candidateAnswers
    ): array
}
```

**Assessment Question Mix (per generation):**
- 30% — Concepts directly from the YouTube course transcript
- 40% — Industry-standard knowledge for the skill category
- 30% — Real-world scenario/practical problems

**Question Types:**
- Short explanation (e.g. "Explain what this code does and why it would fail")
- Debug/fix task (e.g. "Here is a broken Laravel migration — identify and fix the error")
- Scenario judgment (e.g. "A client reports X bug — what is your diagnosis and approach?")
- No multiple choice in core assessment

---

### 3.3 ProctoringService

Collects and evaluates anti-cheat signals from the browser.

```php
class ProctoringService
{
    // Log a signal event during assessment
    public function logEvent(int $sessionId, string $eventType): void

    // Evaluate session signals and return suspicion score
    public function evaluate(AssessmentSession $session): array

    // Flag session for manual review if threshold exceeded
    public function flagIfSuspicious(AssessmentSession $session): void
}
```

**Signals tracked (via Alpine.js frontend → API endpoint):**
- `tab_hidden` — document.visibilitychange fires
- `window_blur` — window.onblur fires
- `copy_paste` — paste event on any answer field
- `answer_time` — time between question render and first keystroke
- `rapid_submission` — answer submitted under 10 seconds

**Suspicion threshold:** 3+ tab switches OR 2+ paste events → flag for review. Does not auto-fail — human review layer.

---

### 3.4 CertenizeService

Handles all communication with Certenize for SBT issuance.

```php
class CertenizeService
{
    // Mint SBT for a passed candidate
    public function mint(
        User $user,
        string $skillName,
        AssessmentSession $session
    ): array  // returns { token_id, tx_hash, credential_url }

    // Verify a credential by token ID
    public function verify(string $tokenId): array

    // Revoke a credential (in case of discovered fraud)
    public function revoke(string $tokenId): bool
}
```

## 4. Assessment Generation Flow (Queue Job)

```
HTTP POST /assessments/generate
        ↓
CourseController stores course record (status: pending)
        ↓
Dispatches GenerateAssessmentJob to queue
        ↓
        [QUEUE WORKER]
        ↓
TranscriptService::fetch() → clean transcript
        ↓
AIAssessmentService::extractConcepts() → concept array
        ↓
AIAssessmentService::generateQuestions() → question set
        ↓
AIAssessmentService::generateRubric() → rubric
        ↓
Assessment record updated (status: ready)
        ↓
Candidate notified via email/in-app
        ↓
Candidate starts assessment session
```

## 5. Prompt Architecture (Claude API)

### Concept Extraction Prompt
```
System: You are a technical curriculum analyst. Extract the core technical 
concepts, tools, and skills taught in the following course transcript. 
Return ONLY a JSON array of concept objects with keys: 
concept, description, difficulty (beginner/intermediate/advanced).

User: [cleaned transcript]
```

### Question Generation Prompt
```
System: You are a senior technical interviewer creating a skills assessment.
Generate [n] assessment questions using this exact distribution:
- 3 questions testing these specific course concepts: [concepts array]
- 4 questions testing general [skill_category] industry knowledge
- 3 real-world scenario problems requiring practical judgment

Rules:
- No multiple choice. All questions require written explanation or code.
- Scenario questions must describe a realistic client/work situation.
- Difficulty should be intermediate level.
Return ONLY a JSON array of question objects with keys:
question_text, type (concept/industry/scenario), expected_length (short/medium/long)

User: Skill category: [category]. Course concepts: [concepts]
```

### Answer Evaluation Prompt
```
System: You are a senior technical assessor. Evaluate the candidate's answers 
against the rubric provided. Be fair but rigorous.
For each answer return a score 0-100 and brief feedback (max 2 sentences).
Return ONLY a JSON array with keys: question_index, score, feedback.

User: Questions: [questions] | Rubric: [rubric] | Answers: [answers]
```

## 7. Frontend Architecture

- **Blade templates** for all server-rendered pages
- **Alpine.js** for:
  - Real-time countdown timer
  - Proctoring event listeners (tab switch, blur, paste)
  - Question navigation (next/prev without page reload)
  - Auto-save draft answers to localStorage every 30 seconds
- **TailwindCSS** for all styling
- **No SPA framework** — server-rendered with progressive enhancement

## 9. Security Considerations

- All assessment answers encrypted at rest (Laravel encryption)
- Wallet addresses never stored in plaintext logs
- Rate limiting on all API endpoints (60 req/min per user)
- Assessment sessions expire after time_limit + 10 minute grace
- NDPR-compliant data handling — candidates can request deletion
- No assessment content cached publicly (questions are per-candidate)
